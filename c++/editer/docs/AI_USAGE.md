## AI 的优势

### 1️⃣ 见多识广 - 全局视野

AI 基于庞大的知识库，能够根据模糊需求给出工具框架。

**实际案例**：

- **文件树需求**: 推荐使用 C++20 标准 + `filesystem` 库，大幅简化开发
- **项目重构**: 指导分解臃肿繁琐的类，实现职责单一原则
- **系统设计**: 主动提议添加 Logger、Result 类型等基础设施

**优势总结**：
- 掌握先进工具的全局概貌
- 快速定位设计方向
- 提供架构层面的优化建议

### 2️⃣ 框架设计能力 - 开箱即用

AI 见过大量优秀代码，擅长搭建标准化框架。

**实际案例（举例）**：

| 框架类型 | 工具 | 由谁完成 |
|---------|------|---------|
| 测试框架 | GoogleTest 风格的自定义框架 | AI 搭建，人填空 |
| 构建脚本 | CMakeLists.txt | AI 编写 |
| 批处理 | build_msvc.bat | AI 编写 |
| 文件树模块 | 类结构 + 接口定义 | AI 搭建，人实现 |

**特点**：
- ✅ 架构清晰，易于扩展
- ✅ 遵循行业最佳实践
- ✅ 减少重复轮子的开发

### 3️⃣ 快速输出 - 高效迭代

代码完成后，能快速生成相关文档。

**产出速度**：
- 类图 → 分钟级
- 架构文档 → 分钟级
- 测试框架 → 分钟级

## AI 的局限

### ❌ 变通能力不足

AI 倾向于"教科书解决方案"，对野路子优化不敏感。

**渲染器 Debug 案例**：

| 阶段 | 做法 | 结果 |
|-----|------|------|
| 第一步 | AI 添加多行代码 | ❌ 问题依旧 |
| 第二步 | 人查看运行效果 | 🔍 发现根本原因 |
| 第三步 | 人添加一个 `refresh()` 调用 | ✅ 问题解决 |
| 结果 | AI 学习这个想法 | 理解并优化 |

**配置解释器案例**：

- **AI 方案**: 构建完整的词法分析、语法分析
- **人的方案**: 使用 `string::split()` 按单词分割（简化问题）
- **实际效果**: 人的方案更快上线

**结论**：
- AI 擅长正规、标准的解决方案
- 对"快速脏实现然后迭代"不敏感

### 🔄 缺乏调试直觉

面对运行时问题时，AI 的调试效率较低。

- 需要反复描述问题才能定位
- 容易过度设计解决方案

- **图形设计**: 了解软件图标的制作工具和流程
- **吃瓜：发现 Windows 路径 BUG**: 路径中的 `+` 符号在 cmd 中的特殊处理

### 🐛 Windows 路径包含 "+" 的问题

**问题描述**：

在 Windows 命令提示符（cmd.exe）中执行包含 `+` 符号的路径时会出现异常。

**具体表现**：
```batch
cd f:\code\c++\editer\test
REM 错误: 无法正确识别路径
```

**原因分析**：

| 层级 | 问题 |
|-----|------|
| **cmd.exe** | `+` 在某些上下文被视为特殊字符或连接符 |
| **路径扩展** | 包含 `+` 的路径在参数扩展时可能被误解 |
| **变量替换** | 在变量名中使用 `+` 路径时容易产生歧义 |

**实际案例**（项目中遇到的）：

```batch
REM ❌ 失败：包含 c++ 的路径
cmake -S F:\code\c++\editer\test -B F:\code\c++\editer\build
REM 错误信息：源目录 "F:/code/c++/editer/test -B F:/code/c++editer/build"...

REM ✅ 解决方案：使用引号包裹路径
cmake -S "F:\code\c++\editer\test" -B "F:\code\c++\editer\build"
```

**解决方案**：

1. **始终为路径添加引号**
   ```batch
   cd "%~dp0"
   cmake -S "%SOURCE_DIR%" -B "%BUILD_DIR%"
   ```

2. **使用相对路径避免**
   ```batch
   cd f:\code\c++\editer\test
   cmake -S . -B build
   ```

3. **使用 PowerShell 替代（更稳定）**
   ```powershell
   cd f:\code\c++\editer\test
   cmake -S . -B build
   ```

**为什么很少人提及**：
- ❌ 这是 cmd.exe 的历史遗留问题
- ❌ 很多开发者用 PowerShell 或 WSL，绕过了这个问题
- ❌ 官方文档很少提及这个坑
- ✅ 在跨平台开发中容易被发现

**预防建议**：
- 项目路径避免使用 `+` 或其他特殊字符
- 在批处理脚本中，所有路径变量都用引号包裹
- 优先使用 PowerShell 或 bash（WSL）



### 💡 开发效率

- ✅ 减少框架设计的时间成本
- ✅ 快速建立项目的基础设施
- ✅ 专注于核心逻辑而非样板代码

## 编程能力培养建议

### 核心理念：广度 + 必要的深度

```
┌─────────────────────────────────────────────┐
│         编程能力结构                        │
├─────────────────────────────────────────────┤
│                                             │
│  广度（AI + 海量信息）    深度（自己掌握）  │
│  ┌──────────────────┐   ┌──────────────┐  │
│  │ 知道有什么工具   │   │ 熟练掌握一门 │  │
│  │ 怎么找到信息     │   │ 或多门语言   │  │
│  │ 框架/库的应用    │   │ 能独立解决   │  │
│  │                  │   │ 核心问题     │  │
│  └──────────────────┘   └──────────────┘  │
│                                             │
└─────────────────────────────────────────────┘
```

### 📡 广度 - AI 的强项

**AI 的本质**：
- 基于庞大的向量化知识库
- 通过基向量组合进行创作
- 将结果投影到需要的向量子空间

**优势**：
- 能有效组织和查询海量信息
- 提供多个角度的解决方案
- 弥补"词条不足"的问题

**使用建议**：
```
你的瓶颈：不知道词条 → 不知道怎么搜索
AI 的优势：提供词条 → 教你怎么搜索
结果：效率大幅提升
```

**例子**：
- 问题：如何处理大文件？
- AI：告诉你"流式处理、内存映射、分块读取"等词条
- 有了词条后，你能精准搜索和学习

### 🦵 深度 - 自己的基础

**交通工具类比**：
```
飞机/高铁（广度+AI）  能带你去大多数地方
         ↓ 当交通工具抛锚时
双腿（深度/扎实基础）  也能往前走，虽然慢但在 running
```

**必要的深度包括**：
- 熟练掌握 1-2 门编程语言
- 理解基础算法和数据结构
- 能独立调试和优化代码
- 理解系统的工作原理

**为什么必要**：
- 服务器瘫痪、网络中断时，AI 不可用
- 对于 AI 无法解决的边界问题，需要自己深入
- 评判 AI 建议的质量需要基础知识
- 野路子优化需要对语言的深刻理解

### 🎯 实践建议

| 阶段 | 策略 | 目标 |
|-----|------|------|
| **初期** | 依赖 AI 框架 + 人填空 | 快速上线，累积经验 |
| **中期** | 开始质疑 AI 方案 | 培养评判能力 |
| **后期** | 组合 AI 广度 + 自己深度 | 高效且有品质的输出 |

**关键是**：
1. ✅ 认识到 AI 和自己各自的优势
2. ✅ 在合适的地方用合适的工具
3. ✅ 不断挑战 AI 的方案，积累深度
4. ✅ 最终形成"见多识广"且"功夫深"的工程师

## 总结

使用 AI 开发的最大收获不是快速完成项目，而是：

- 📚 **开阔视野** - 了解更多工具和方法论
- 🔨 **学习框架** - 见识优秀代码的组织方式
- 💪 **提升深度** - 通过批评 AI、优化 AI 方案来强化自己的理解
- ⚡ **提高效率** - 专注核心逻辑，减少样板代码

**最终目标**：
> 做一个既有 AI 时代的广度，又有传统工程师深度的社区“划水员”。
